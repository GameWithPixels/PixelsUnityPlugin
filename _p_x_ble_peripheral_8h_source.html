26 // We can't find any reliable information about the thread safety of CoreBluetooth APIs
27 // so we're going to assume that CBCentralManager, CBPeripheral, CBCharacteristic and CBDescriptor
28 // achieve any required synchronization with the given queue
29 // When no queue is given to the central manager, it defaults to the main thread serial queue.
30 // We create a serial queue as well so we don't have to worry about synchronization when we use
31 // the queue ourselves (and obviously CoreBluetooth works well with a serial queue).
32 // We're not concerned about performance by using a serial queue as Bluetooth LE operations
33 // are "low" frequency by design anyways.

69 // This queues BLE operations (requests)
70 // The connect request will wait indefinitely until the connection is made, and will block
71 // any further request until connection. To cancel a blocking connection call cancelQueue().
72 // A request keep a strong reference to the instance so the later won't be de-allocated until the queue is empty.
73 // On being de-allocated, instance will cancel connection to peripheral
74 // Handlers (such as request completion handler) are called on the shared BLE queue,
75 // user code for those handlers should return as quickly as possible to avoid blocking/delaying any other BLE event.
